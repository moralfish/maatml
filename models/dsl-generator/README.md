# DSL Generator

Local generative model that translates text flow descriptions into
canonical Flow DSL documents. Powers Flow Studio's "Generate" mode.

- **Task string (manifest):** `dsl_generation`
- **Model id:** `dsl-generator:v1`
- **Base model:** Qwen/Qwen2.5-Coder-7B-Instruct (LoRA fine-tune via PEFT)
- **Runtime backend:** `CandleGenerativeBackend`
- **Response schema:** `{ "dsl": "<canonical DSL text>" }`


The generative backend itself is generic Llama/Qwen-style inference; only the
`prompt_spec.json` packed alongside the weights is task-specific.

- The Models drawer can filter by task.
- The Generate pane lists only DSL-capable models.
- Each task evolves its prompt spec, schema, and decoding params independently.


## Dataset

- `datasets/prompt_spec.json` - system prompt teaches the DSL grammar (header,
  node syntax with the four kinds, edge syntax with three outcomes), one inline
  few-shot example, decoding params (`temperature: 0.0`, `max_new_tokens: 512`).
- `datasets/schema.json` - JSON schema for the on-disk training sample
  (`sample_id`, `source`, `description`, `dsl`, `split`).
- `datasets/grammar.md` - human-readable DSL reference, mirrors what
  `crates/flow-dsl/src/serializer.rs` produces.
- `datasets/samples/seed_samples.jsonl` - 92 hand-crafted `{description, dsl}`
  pairs covering all four node kinds, all five action adapters (`mock`, `zowe`,
  `zosmf`, `ssh`, `shell`), all three edge outcomes (`pass`, `fail`, `always`),
  single-node up to multi-branch flows.
- `datasets/samples/eval_samples.jsonl` - 100 hand-written held-out evaluation
  samples that do not appear in the training pool.
- `datasets/samples/augmented_samples.jsonl` - generated by `flow_ml prepare` via the rule-based augmenter; gitignored.

## End-to-end commands

All commands below take this model folder as their primary argument. Outputs
land under `output/` (gitignored).

```bash
# 1. Prepare splits (runs the augmenter, then 60/20/20 hash-stable split)
python -m flow_ml.cli prepare models/dsl-generator/

# 2. Smoke training (SmolLM2-135M, 6 steps)
python -m flow_ml.cli train models/dsl-generator/ --smoke

# 3. Full training (Qwen2.5-Coder-7B, 3 epochs)
python -m flow_ml.cli train models/dsl-generator/

# 4. Evaluate the most recent checkpoint
python -m flow_ml.cli evaluate models/dsl-generator/

# 5. Package -> f16 .fm archive
python -m flow_ml.cli package models/dsl-generator/ --version v1

# 6. Verify the .fm
python -m flow_ml.cli verify models/dsl-generator/output/dist/dsl-generator-v1.fm
```

## Evaluation criteria

`evaluate_dsl` computes:

- **`json_validity`** - fraction of generations that parse as JSON with a
  string-valued `dsl` field.
- **`parser_roundtrip`** - fraction of generated DSL strings that the real
  Rust `flow_dsl::parse` accepts (via the `flow_dsl_py` PyO3 binding from
  `flow-studio/crates/flow-dsl-py`). When the binding is not installed, the
  fallback is a heuristic check that the DSL has a `flow "..."` header.
- **`node_count_jaccard`** - sample-wise Jaccard of node IDs between gold and
  prediction; not a strict equality (the model is allowed to choose its own
  node IDs as long as the structure is right).
- **`edge_count_match`** - exact match on the number of edges.

Reports land in `output/eval/<run-name>.{json,md}`.

## Inputs/outputs at runtime (Candle)

The runtime sends a single ChatML turn:
- system: contents of `prompt_spec.json`'s `system` field.
- user: `prompt_spec.json`'s `user_template` with `<<USER_DESCRIPTION>>` substituted.
